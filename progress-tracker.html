🗄️ Day 3 — Entity Framework Core + SQL Server Database Setup

📌 What I Built Today

  • ✅ Installed EF Core NuGet packages — Microsoft.EntityFrameworkCore.SqlServer v10.0.9 and Microsoft.EntityFrameworkCore.Tools v10.0.9
  • ✅ Created AppDbContext — inherits DbContext, exposes DbSets for all 5 modules (JournalEntries, BankTransactions, Invoices, Grants, Budgets)
  • ✅ Configured SQL Server LocalDB connection string in appsettings.json (FundFlowNXT database)
  • ✅ Registered AppDbContext in Program.cs with UseSqlServer — dependency injection wired up
  • ✅ Generated InitialCreate migration — creates 5 tables with correct column types and identity PKs
  • ✅ Migrated JournalEntryController from in-memory static list to real EF Core database
  • ✅ Injected AppDbContext via constructor — replaced hardcoded seed data with DB reads
  • ✅ GetAll → ToListAsync() | GetById → FindAsync() | Create → Add() + SaveChangesAsync()
  • ✅ Made all controller actions async/await with Task<ActionResult<T>>
  • ✅ Migrated TreasuryController — replaced static BankTransactions list with AppDbContext; added POST Create endpoint
  • ✅ Migrated AccountsPayableController — replaced static Invoices list with AppDbContext; all workflow endpoints (approve/pay) now persist to DB
  • ✅ Created DbSeeder.cs — seeds all 5 tables on first startup if empty (3 journal entries, 5 bank transactions, 4 invoices, 3 grants, 3 budgets)
  • ✅ Wired DbSeeder into Program.cs — runs at app startup using a scoped service scope
  • ✅ Migrated GrantsController — replaced static list with AppDbContext; added POST Create; summary and spend endpoints now read/write to DB
  • ✅ Migrated BudgetingController — replaced static list with AppDbContext; comparison and summary now query live DB data
  • ✅ All 5 controllers (GL, Treasury, AP, Grants, Budgeting) fully connected to SQL Server — backend 100% on the database

🧠 Key Learnings

  • ✔ Entity Framework Core — ORM that maps C# classes to SQL tables, eliminates raw SQL for CRUD
  • ✔ DbContext — the bridge between your app and the database; DbSet<T> represents a table
  • ✔ Dependency Injection — DbContext is registered as a service in Program.cs and injected into controllers via constructor
  • ✔ EF Migrations — dotnet ef migrations add creates a versioned snapshot of schema changes; update-database applies them
  • ✔ async/await — database calls are I/O-bound so they must be awaited; Task<ActionResult<T>> is the async return type
  • ✔ Connection strings — Server, Database, authentication mode configured in appsettings.json; read at runtime via IConfiguration
  • ✔ FindAsync vs FirstOrDefaultAsync — FindAsync looks up by primary key and is faster for single-row lookups
  • ✔ Seed data pattern — check if table is empty before inserting; use SaveChanges() once at the end, not per row
  • ✔ IServiceScope — DbContext is scoped (one per request), so seeding at startup needs a manual scope via app.Services.CreateScope()

⚠️ Challenges Faced

  • ❌ Computed properties (IsOverspent, IsBalanced, etc.) can't be mapped to database columns — EF only persists stored fields
  • ✅ Kept computed properties on the model as C# getters; EF ignores them automatically
  • ❌ DbContext is scoped — can't be used directly from the root service provider at startup
  • ✅ Wrapped seed call in app.Services.CreateScope() to get a properly scoped DbContext instance

🚀 Next Steps

  • ➡ Start Angular frontend — components, routing, HttpClient service to call the API
  • ➡ Build journal entry list page and wire it to the GL API
  • ➡ Plan Azure deployment — App Service + Azure SQL

Completed the full database migration — all 5 controllers (GL, Treasury, AP, Grants, Budgeting) are now reading and writing to SQL Server via EF Core. AppDbContext, InitialCreate migration, and DbSeeder all in place. The entire backend is persistent, async, and production-ready from a data layer perspective.

📌 What I Built Today

  • ✅ Scaffolded Angular 19 project — fundflow-client in frontend/ folder
  • ✅ Configured app.config.ts — registered provideHttpClient() and provideRouter() for standalone app
  • ✅ Built ApiService (src/app/services/api.ts) — HttpClient-based service with methods for all 5 modules
  • ✅ getJournalEntries() → GET /api/JournalEntry
  • ✅ getBankTransactions() → GET /api/Treasury

  • ✅ Built app.html template — loading spinner state + data table with columns: ID, Description, Debit, Credit, Fund, Balanced
  • ✅ Balanced column uses class binding [class.balanced] / [class.unbalanced] for visual indicator

🧠 Key Learnings

  • ✔ Angular standalone components — no NgModule needed; imports declared directly on @Component
  • ✔ provideHttpClient() — registers Angular's HTTP layer in app.config.ts instead of importing HttpClientModule
  • ✔ Injectable service — @Injectable({ providedIn: 'root' }) makes ApiService a singleton available app-wide
  • ✔ Observable + subscribe — HttpClient.get() returns an Observable; subscribe() fires when data arrives (next) or fails (error)
  • ✔ ngOnInit — lifecycle hook that runs after component is constructed; right place to fetch data
  • ✔ *ngIf / *ngFor — structural directives to conditionally render and loop over data in the template
  • ✔ Class binding [class.name] — conditionally applies a CSS class based on a boolean expression
  • ✔ ChangeDetectorRef — manually triggers Angular's change detection after async data arrives

  • ✅ CORS already configured in Program.cs from Day 1 — just needed to verify the Angular dev server port matched
